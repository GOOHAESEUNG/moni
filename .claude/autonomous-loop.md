# 자율 루프 지침

## 목표
KIT 바이브코딩 공모전 심사 기준 40점 만점 달성. 매 라운드마다 평가 → 최저 점수 항목 수정 → 재평가.

## 라운드별 진행

### 매 라운드 실행 순서
1. `docs/judging-criteria.md` 읽기
2. 현재 상태 **냉정하게** 평가 (각 항목 점수 + 근거)
3. 가장 낮은 점수 영역 파악
4. 해당 영역 수정 작업 실행 (**개발 파이프라인** 준수)
5. `npm run build` 통과 확인
6. 데모 동선 스모크 테스트 (주요 링크 클릭 가능 여부)
7. **스모크 테스트 실패 시 push 금지** → 수정 후 재시도
8. `git add + commit + push`
9. `docs/judging-criteria.md`에 점수 업데이트
10. 같은 에러 2회 반복 시 해당 작업 중단 → 다음 항목으로

### 라운드 출력 형식 (매 라운드 끝에 기록)
```
## 라운드 N 결과
- 목표: [이번 라운드에서 수정한 항목 1개]
- 점수 변화: [X점 → Y점, 근거 3줄]
- 수정 파일: [파일 목록]
- 빌드: ✅/❌
- 데모 검증: ✅/❌
- 다음 라운드 금지사항: [이번에 실패한 접근 방식]
- 총점: [현재 총점/40]
```

### 점수 평가 원칙 (극도로 냉정하게)
- **심사위원 관점**으로만 평가. 개발자 관점 금지.
- "잘 만들었다"가 아니라 "심사위원이 3분 체험했을 때 실제로 느끼는 인상" 기준.
- **희망적 점수 금지**: "이 정도면 됐겠지" 식의 관대한 채점 절대 금지.
- **증거 기반**: 점수를 줄 때 반드시 구체적 근거를 함께 적기. "잘 됨" 같은 모호한 평가 금지.
- **0점도 줘야 하면 준다**: 기능이 없거나 완전히 깨져있으면 0점.
- **만점은 정말 완벽할 때만**: 배점 2점 항목에서 2점은 "경쟁작 대비 상위 10%" 수준일 때만.
- **비교 기준**: "대학생 해커톤 출품작 중 상위권"이 아니라, "실제 서비스로 출시해도 되는 수준"으로 평가.
- **첫인상 테스트**: 각 페이지를 처음 보는 사람이 5초 안에 "이게 뭔지" 이해하는가?
- **데모 중 에러 한 번이면 감점**: 심사위원 앞에서 에러가 나면 신뢰도 급락.

### 작업 우선순위
1. **README.md 전면 재작성** — 현재 create-next-app 기본 템플릿. 심사위원 첫인상 결정
2. **데모 플로우 E2E 검증** — 모든 링크/버튼 클릭 동작 확인
3. **.env.example 생성** — 심사위원이 로컬에서 실행할 수 있도록
4. **UI/UX 일관성** — 버튼 크기, 간격, 색상 통일
5. **제출 문서 보강** — submit-draft.md 완성 (시연 시나리오, 화면 설명, 한계점)
6. **학술 근거 최신화** — 2024~2026 논문 추가
7. **불필요 파일 정리** — ia-board.html 등 제거
8. **디자인 폴리시** — impeccable /critique + /audit

### 개발 파이프라인 (반드시 준수)
```
1. planner (계획 수립) — 뭘 할지 스펙 정리
2. coder/codex (구현) — Codex 서브에이전트에게 코드 작성 위임
3. design-reviewer (UI 검수) — Skill 도구로 impeccable:critique + impeccable:audit 실행
4. reviewer/codex (코드 리뷰) — Codex에게 diff 품질 검토
5. 스모크 테스트 — 데모 동선 링크/버튼 동작 확인
6. commit + push (스모크 테스트 통과 후에만)
```
